## 项目说明

本项目将易语言 `.e` 工程文件解包为可读目录，或将目录回包为 `.e` 文件。

让易语言项目享有 Git 版本管理、代码 Diff、AI 辅助编辑等现代开发体验。

### 实现目标

对易语言.e源代码文件进行拆包为文件夹，支持从文件夹再封装回.e，支持修改txt中的代码，支持新增加程序集、类、模块引用等操作后进行再封装，在对代码文件夹封包时，必须为语义解析优先，避免各种难看的兜底写法。

### 图片、音频及二进制资源逻辑

* 解包后的图片与音频资源本质上都是易语言常量资源，使用方式和普通常量一致，代码中以 `#资源名` 引用。
* `image/` 与 `audio/` 目录只是按易语言资源页类型分组；除图片、音频外，也可存储程序运行需要携带的任意二进制数据。
* `image/list.json` 与 `audio/list.json` 是资源索引，条目中的 `logicalName` 是常量名，`relativePath` 指向实际二进制文件，`key` 用于维持封包时的稳定内部映射。
* 新增资源应优先使用 `e-packager update <目录> --add-image <文件>` 或 `--add-audio <文件>`，可写成 `资源名=文件路径` 显式指定常量名；不要只把文件丢进目录而不更新 `list.json`。
* 新增资源会改变最终.e的常量资源表，必须让封包流程走语义重建，避免复用旧原生快照导致新增资源丢失。

### 项目规范

* 发现存在连续打补丁修复问题时，应仔细思考是否实现方向上有设计上或系统性错误，避免形成屎山代码。
* 项目的依赖要精简，避免引入vcpkg包。
* 注重项目的可维护性，对于可独立工作的工具类、管理类，应单独封装。
* 如果某个文件大于3000行，应考虑复用性及可维护性进行合理的拆分。
* 避免单文件代码量过于庞大难以维护。
* 必要时使用IDA-PRO的MCP来对易语言主EXE进行逆向来实现功能。
* 对.h头文件始终添加简单的中文注释。
* 源代码文件保持UTF8+BOM的格式保存，注意中文是否存在乱码并进行修正。
* 源代码文件使用windows下的换行符（CRLF）。
* 项目使用ISO C++20 标准 (/std:c++20)，可使用其特性开发。

### 编译

请在完成修改后验证release+x86、release+x64是否可正常编译。

请参考此命令行方法进行编译，不同设备和VS版本的MSBuild.exe的位置可能不同，请注意先寻找及确认。

```powershell
MSBuild.exe ..\\e-packager.vcxproj /t:Build "/p:Configuration=release;Platform=Win32" /m
```

如果编译存在报错，请进行修正，确保功能实现正常的情况下可正常编译。

### 测试

**只有强调需要进行全量测试时才需要执行，不要每次修改都执行**

使用win32\x64两个版本的exe分别测试，将“C:\Users\aiqin\OneDrive\code\模块”中所有的.e文件（不包含子目录）进行拆包再回包，确保回包后字节与MD5与原文件相同，不要改动原文件，而是将其复制一份后在temp目录中进行测试，如果不通过，请查找原因并进行修复，最终直到问题被彻底修复。

如果文件比较小，那也不一定是有问题的文件，但文件尺寸为0，那一定是坏文件，请跳过测试。
